# Плохой докерфайл:
``` bush
FROM ubuntu:latest
FROM python:latest

ENV testfile test.sh

RUN apt-get update -y > /dev/null && apt-get upgrade -y > /dev/null
RUN apt-get install bash neofetch -y > /dev/null
ADD . .
RUN bash $testfile
RUN python3 test.py
```
# В чем проблемы?

1. Больше чем один FROM - при этом используется только второй
2. :latest - image может сломаться в новой обнове, поэтому лучше специфицировать версию
apt-get
3. ungrade - примерно тоже самое что и предыдущий пункт
4. apt-get unpdate не в одной строчке с apt-get install - тогда update закэшируется и не будет запускаться каждый билд
5. ENV - нужно объявлять его как можно позже
6. ADD - разархивирует архивы и wget-ит ссылки, что может быть проблемой
7. Запуск чего-то во во время билда - это плохо, так как image должен быть многоразовым, а так этот принцип нарушается

## 2 Часть 

# Плохой docker-compose.bad.yml
# В чем проблемы?
1. :latest теги — образы могут измениться, сборка станет невоспроизводимой, проект может сломаться после обновления
2. Хардкод секретов — пароли попадают в репозиторий, риск утечки, сложно менять без правки кода
3. Избыточные порты — БД доступна извне, это увеличивает поверхность атаки
4. Нет healthcheck — сервисы стартуют раньше зависимостей, приложение падает с ошибкой
5. privileged: true — контейнер получает полный доступ к хосту, риск компрометации


# Хороший docker-compose.good.yml:
Заменить :latest на конкретные версии с alpine, вынести секреты в .env через переменные, для внутренних сервисов использовать expose вместо ports, добавить healthcheck и condition: service_healthy, убрать privileged, добавить лимиты ресурсов и restart policy, разделить сервисы по разным сетям с internal: true для БД.

В результате имеем исправление версий образов, вынос секретов в .env и закрытие лишних портов сделали сборку предсказуемой и безопасной. Настройка healthcheck, лимитов ресурсов и политики перезапуска обеспечила стабильную работу и изоляцию контейнеров.
